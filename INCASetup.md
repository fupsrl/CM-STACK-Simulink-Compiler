# INCA Setup Guide

This guide walks through importing the CM-STACK artifacts into INCA and connecting to the XCP target running on Raspberry Pi.

---

## Prerequisites

- The workflow has completed successfully (`run_native_xcp_workflow` returned without errors).
- The Raspberry Pi has a **static IP address** assigned. On Raspberry Pi OS Lite (Trixie and later) no static IP is configured by default — set one via `nmcli` or `/etc/dhcpcd.conf` before proceeding.
- INCA is installed with the XCP-over-Ethernet option licensed.

---

## 1. Build output overview

After a successful run the following artifacts are available under `workflow/artifacts/<ModelName>/`:

| File | Description |
|---|---|
| `<model>_inca.a2l` | **Patched A2L — use this in INCA** |
| `<model>.a2l` | Vanilla A2L generated by XCPlite (use in CANape) |
| `<model>.hex` | INCA-compatible Intel HEX |
| `<model>.s19` | INCA-compatible Motorola S19 |
| `<model>_raw.hex` | Raw HEX directly from the ELF |
| `XCP_104.aml` | XCP protocol layer description (Vector XCPlite) |

## 2. Create a new INCA project / open existing one

Open INCA and create a new experiment or open an existing project where you want to add the Raspberry Pi target.

### 2.1 Import the A2L and HEX

When INCA prompts for a description file, select the **`_inca.a2l`** file. At the same step, also import the **`.hex`** file so INCA can read the calibration page contents.

<img width="608" height="463" alt="image" src="https://github.com/user-attachments/assets/44a9afd2-94dc-4855-b65d-749546832699" />

> **Note:** always use `_inca.a2l`, not the vanilla one. The patched file contains corrected memory segments, display formats, INCA-compatible event names, and the EPK string address required for HEX validation.

## 3. Create a new XCP device

In the INCA Hardware Configuration, add a new device of type **XCP over Ethernet (TCP)**.

<img width="260" height="135" alt="image" src="https://github.com/user-attachments/assets/8c358d2d-aeef-4ffe-9cfc-19f534591ac9" />

Assign the A2L description file to the device:

<img width="560" height="192" alt="image" src="https://github.com/user-attachments/assets/f9f5820e-0f9c-48c6-9eda-d6beb55900d6" />

Configure the connection parameters to match the Raspberry Pi:

| Parameter | Value |
|---|---|
| Transport | UDP |
| IP address | Raspberry Pi static IP (e.g. `192.168.1.10`) |
| Port | `5555` (default, configurable via `XcpPort`) |

## 5. Measurement & calibration

Once the device is configured and the executable is running on the Raspberry Pi, open the experiment in INCA. Signals marked as **Test Points** in Simulink appear as measurements; parameters with **Tunable** storage class appear as calibration objects.

<img width="1530" height="887" alt="image" src="https://github.com/user-attachments/assets/76e713e4-bcc7-4cd0-8a1d-bfdfb31c25f4" />

- Start the XCP session to begin live measurement.
- Modify calibration values online — changes are reflected on the target immediately.
- Save the calibration to a CDFX file for reproducibility.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| INCA cannot connect | Wrong IP or port | Verify `RaspberryHost` and `XcpPort` match the device config |
| HEX validation fails | Wrong HEX imported | Re-import `_inca.hex` (not `_raw.hex`) |
| No measurements visible | Signals not marked as Test Points | In Simulink: right-click signal → Properties → ☑ Test Point |
| No calibration objects | Default parameter behavior not set to Tunable | Model config → Code Generation → Optimization → Default parameter behavior: Tunable |
| A2L import error in INCA | Using vanilla A2L | Use `_inca.a2l`, not the plain `.a2l` |
